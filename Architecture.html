4.1. Декомпозиция на компоненты и обязанности

Архитектура платформы Dockies строится по принципу модульного монолита с чётким выделением доменных сервисов. Это позволяет сохранить высокую производительность и простоту развёртывания, присущие монолиту, но при этом обеспечить логическую изоляцию функциональных областей.

КомпонентОбязанности
Сервер приложений (API Gateway)Единая точка входа для клиентских приложений. Аутентификация и авторизация запросов по токену. Маршрутизация запросов к внутренним сервисам.
Сервис документовУправление жизненным циклом документа: создание, обновление атрибутов (тип, статус, контрагент, сумма). Взаимодействие с репозиторием для сохранения файлов. Контроль версий.
Репозиторий документовХранение файлов всех версий, вычисление и проверка контрольных сумм (SHA-256), обнаружение конфликтов версий. Реализован на объектном хранилище (S3).
Сервис маршрутизацииОпределение маршрута согласования по типу документа и настроенному шаблону. Формирование и передача задач между участниками, контроль сроков, отправка напоминаний и эскалаций. Поддержка параллельных маршрутов и сценариев коллегиального согласования.
Сервис аутентификацииИнтеграция с корпоративной службой каталогов (AD) для проверки учётных данных. Выдача и проверка токенов сессии (JWT). Реализация режима ручной аутентификации при недоступности AD.
Справочник оргструктурыКэшированная база сотрудников, ролей, иерархии и замещений. Периодическая синхронизация з HRIS (один раз в сутки). Предоставление данных сервису маршрутизации и API.
Сервис электронной подписиВзаимодействие с криптопровайдером (УЦ) для наложения КЭП и УНЭП. Проверка статуса сертификата через протокол OCSP. Управление очередью отложенного подписания при недоступности провайдера.
Сервис уведомленийОтправка уведомлений по email, push и SMS через внешний шлюз. Переключение на альтернативный канал при недоступности основного. Поддержка очереди повторных попыток.
Журнал аудитаЗапись всех значимых событий в защищённую схему БД с формированием криптографической цепочки (хеш предыдущей записи). Проверка целостности, экспорт акта аудита. Защита от модификации и удаления записей на уровне прав доступа.
Сервис ЭДООтправка подписанных документов контрагенту через внешнего оператора ЭДО (синхронно с очередью). Приём встречной подписи и протоколов разногласий через веб-хуки.
Сервис проверки лимитовЗапрос бюджетных лимитов и статуса контрагента из ERP (синхронно, таймаут 3 секунды). Поддержка ручного режима подтверждения лимитов при недоступности ERP.
Веб-приложение (Desktop UI)Пользовательский интерфейс для работы через браузер (React). Отображение карточек документов, задач, версий, журнала аудита. Отправка запросов к API Gateway.
Мобильное приложение (Mobile UI)Адаптивная веб-версия (PWA) для мобильных устройств с поддержкой оффлайн-кэширования. Просмотр документов, согласование, подписание УНЭП по SMS-коду.

4.2. Ключевые архитектурные решения (ADR)

ADR-1: Логический монолит с доменными сервисами

Контекст:
Требования к системе включают высокую производительность (NFR‑1, NFR‑2) и поддержку до 500 одновременных пользователей (NFR‑7). Команда разработки небольшая (~5 человек). Необходимо быстро выводить новые функции.

Решение:
Реализовать систему как логический монолит, где компоненты разделены на уровне кода и вызовов, но деплоятся как единое приложение. Каждый доменный сервис (документы, маршрутизация, аудит) изолирован и общается через чёткие интерфейсы.

Отвергнутая альтернатива:
Микросервисная архитектура с независимым деплоем каждого компонента.

КритерийМодульный монолитМикросервисы
Сложность разработкиНизкая — единая кодовая база, простой отладчикВысокая — распределённая трассировка, управление конфигурацией
ПроизводительностьВысокая — вызовы внутри процессаНиже — сетевые вызовы, сериализация
Простота деплояОчень простая — один артефактСложная — оркестрация многих сервисов
Масштабирование частейНе поддерживается — масштабируется только весь монолитГибкое — можно масштабировать только нагруженные сервисы

Последствия:

  • Простота разработки и ускорение вывода функций на рынок.
  • Нулевые накладные расходы на межсервисное взаимодействие, что помогает соблюсти SLA по времени отклика (NFR‑1, NFR‑2).
  • Единая база данных с транзакционной целостностью (SR‑SYS‑30) без сложности распределённых транзакций.
  • При росте нагрузки масштабировать придётся всё приложение целиком (компромисс приемлем для целевых 500 пользователей).

ADR-2: Криптографическая цепочка в реляционной БД для аудита

Контекст:
Требование неизменяемости журнала аудита (SR‑SH‑7) при стандартном стеке технологий (PostgreSQL). Необходимо найти баланс между юридической значимостью (SR‑SH‑12) и стоимостью решения.

Решение:
Хранить журнал аудита в отдельной, защищённой схеме реляционной БД, связывая записи криптографической цепочкой. Каждая новая запись содержит хеш предыдущей (SHA‑256), что позволяет обнаружить факт модификации. Проверка целостности запускается ежечасно.

Отвергнутая альтернатива:
Использование WORM-носителей (однократная запись) или специализированных блокчейн-платформ.

КритерийКриптоцепочка в БДWORM-носитель
СтоимостьНулевая аппаратная — реализуется программноТребует специализированного оборудования
Гибкость развёртыванияРаботает на любой инфраструктуре, включая облачнуюПривязка к физическому носителю
Защита от модификацииОбнаруживает модификацию, но не предотвращает физическиФизически невозможна перезапись
Восстановление данныхНе восстанавливает повреждённые записиДанные не повреждаются

Последствия:

  • Отсутствие зависимости от дорогого оборудования. Платформа может разворачиваться в облаке.
  • Программная проверка целостности в автоматическом режиме.
  • Между модификацией записи и её обнаружением есть временной промежуток (до часа).
  • Обнаружение факта нарушения не восстанавливает данные — требуется резервная копия (инкрементальные копии каждые 4 часа).
  • Прямой доступ к БД журнала строго ограничен на уровне инфраструктуры.

ADR-3: Единая реляционная база для структурированных данных

Контекст:
Платформа хранит карточки документов, задачи, записи аудита, роли и шаблоны. Ожидаемый объём — до 1 000 000 карточек и 5 ТБ файлов в объектном хранилище, до 500 одновременных пользователей.

Решение:
Все структурированные данные (атрибуты, связи, статусы) хранить в единой реляционной БД (PostgreSQL). Журнал аудита — в отдельной схеме с ограниченными правами. Файлы документов — в объектном хранилище (S3). Тяжёлые аналитические запросы направлять на реплику для чтения.

Отвергнутая альтернатива:
Отдельная база для каждого логического модуля или нереляционное хранилище для аудита.

КритерийЕдиная реляционная БДРаздельные хранилища
Согласованность данныхТранзакционная целостность (ACID)Требуются распределённые транзакции
Сложность администрированияОдна база — единая политика бэкаповМножество политик, больше точек отказа
Запросы между модулямиПростые JOIN (например, документ + задача)Сложная агрегация на уровне приложения

Последствия:

  • Транзакционная целостность данных без распределённых транзакций (SR‑SYS‑30).
  • Низкая сложность администрирования и резервного копирования (NFR-8).
  • Простота и скорость запросов между модулями.
  • Потенциальное узкое место: высокая частота записи в аудит может конкурировать за ресурсы (смягчается через реплику).
  • При объёме 1 млн документов нагрузка укладывается в возможности одного сервера.

ADR-4: Асинхронная очередь для взаимодействия с внешними системами

Контекст:
Dockies интегрируется с AD, ERP, HRIS, криптопровайдером, оператором ЭДО и шлюзом уведомлений. Внешние системы могут быть недоступны (NFR-4).

Решение:
Все вызовы к внешним системам, результат которых не требуется пользователю немедленно, помещаются в асинхронную очередь (RabbitMQ) с повторными попытками (retry с exponential backoff). Синхронными остаются только вызовы, результат которых необходим для продолжения операции пользователя (проверка сертификата OCSP, проверка лимитов). Для синхронных вызовов установлен таймаут (3 секунды) с переходом в режим деградации.

Отвергнутая альтернатива:
Все вызовы к внешним системам — синхронные. При недоступности внешней системы операция пользователя блокируется.

КритерийАсинхронная очередьПолностью синхронные вызовы
Время отклика для пользователяБыстрый ответ — операция поставлена в очередьЗависит от времени ответа внешней системы
Поведение при сбое внешней системыОперация сохраняется в очереди, пользователь продолжает работуОперация блокируется или завершается ошибкой
Гарантия доставкиВысокая — очередь с повторами и сохранениемНизкая — при ошибке требуется ручной повтор
Сложность реализацииСредняя — требуется очередь, обработка повторов, идемпотентностьНизкая — простой запрос-ответ

Последствия:

  • Устойчивость к сбоям внешних систем — отправка уведомлений, документов и синхронизация данных не блокируют пользователя и выполняются при восстановлении (NFR-3).
  • Соблюдение таймингов отклика пользовательского интерфейса (NFR-1, NFR-2).
  • Немедленная синхронизация невозможна — информация может появляться с небольшой задержкой (секунды–минуты).
  • Отладка асинхронных цепочек сложнее, но компенсируется централизованным логированием.
  • Идемпотентность достигается уникальными идентификаторами операций.

ADR-5: Адаптивная веб-версия для мобильного доступа

Контекст:
Система должна поддерживать мобильный доступ (SR‑SH‑2, SR‑SH‑8), включая просмотр, согласование и подписание УНЭП по SMS-коду.

Решение:
Основной мобильный интерфейс реализовать как адаптивную веб-версию с PWA и поддержкой оффлайн-кэширования через сервис-воркер. Нативное приложение — опционально на следующем этапе.

Отвергнутая альтернатива:
Разработка нативных приложений для Android и iOS с первого этапа.

КритерийАдаптивная веб-версия (PWA)Нативные приложения
Стоимость разработкиНизкая — единая кодовая базаВысокая — две платформы или кросс-фреймворк
Скорость выходаБыстрая — адаптация интерфейсаМедленная — отдельный цикл разработки
ОбновлениеМгновенное при загрузке страницыЧерез магазин приложений
Оффлайн-возможностиОграниченные (кеширование через сервис-воркер)Полноценные (локальная БД)
Доступ к оборудованиюОграниченныйПолный

Последствия:

  • Быстрый выход на мобильную платформу с низкими затратами.
  • Мгновенные обновления интерфейса для всех пользователей.
  • Возможность согласования и подписания УНЭП через браузер (SR‑SYS‑17).
  • Полноценная работа без сети невозможна, подписание КЭП на мобильном устройстве не поддерживается (требуется ПК).
  • Сценарий, требующий подписания КЭП с мобильного устройства, обрабатывается через статус «Ожидает подпись с ПК» (SR‑SYS‑19).
  • При необходимости функциональность нативных приложений будет добавлена на следующих этапах.
