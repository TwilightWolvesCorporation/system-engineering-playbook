1.1. Описание системы и её границы

Dockies (Documents and Cookies) — корпоративная платформа электронного документооборота и согласования, обеспечивающая:

  • создание и версионирование документов (заявки, договоры, служебные записки);
  • настраиваемые маршруты согласования с контролем сроков;
  • юридически значимый аудит и неотказуемость действий;
  • интеграцию с корпоративной инфраструктурой (AD, справочники, уведомления, хранилище).

Что входит в границы системы

  • Управление жизненным циклом документа (создание → согласование → подписание → исполнение → архив).
  • Разграничение доступа на основе ролей (AD групп).
  • Логирование всех действий с документом (кто, когда, что изменил).
  • Генерация дашбордов и уведомлений о статусах.
  • Настройка маршрутов согласования администратором.

Что не входит в границы системы (требует внешних систем)

  • Долговременное хранение документов (передаётся во внешний архив).
  • Квалифицированная электронная подпись (интеграция с криптопровайдером/УЦ).
  • Формирование бухгалтерских проводок и платежей (интеграция с ERP/банком).
  • Нормативно-справочная информация о штатной структуре (получается из HRIS/AD).

2.3. Сценарий 1. Создание и согласование договора с контрагентом

Цель

Создать проект договора с контрагентом, провести юридическую экспертизу, бизнес‑утверждение, подписать КЭП и направить контрагенту через оператора ЭДО, с контролем версий и полным аудитом.

Условия начала

  • Инициатор авторизован в Dockies через Модуль единого входа – получен токен с ролями.
  • Справочник оргструктуры (HRIS) актуален – известны роли Юриста и Согласующего.
  • Реквизиты контрагента проверены через систему бюджетирования / ERP – статус контрагента активен.
  • Шаблон договора доступен для загрузки.

Основной поток

  1. Инициатор создаёт карточку договора, заполняет реквизиты (контрагент, сумма, предмет, статья расходов) и загружает файл – Версия 1.
  2. Маршрутизатор обращается к Справочнику оргструктуры, определяет маршрут: первый шаг – Юрист.
  3. Юрист получает уведомление через Сервис уведомлений (email и/или push-уведомление), открывает задачу, вносит правки, загружает Версию 1.1 с пометкой «Правки юриста».
    • Репозиторий документов сохраняет обе версии с контрольными суммами, связывая их с одной карточкой.
  4. Юрист нажимает «Согласовано» – система фиксирует визу в журнале аудита и передаёт задачу Согласующему.
  5. Согласующий просматривает список версий (номер версии, автор, дата, комментарий, контрольная сумма), утверждает файл и накладывает КЭП через Модуль электронной подписи:
    • Модуль вычисляет хеш файла, отправляет в криптопровайдер, получает подписанный хеш и штамп времени, сохраняет подпись в карточке документа.
    • Журнал аудита фиксирует: ID подписанта, контрольную сумму, серийный номер сертификата, временную метку.
  6. Модуль интеграции с оператором ЭДО направляет подписанный документ Контрагенту и фиксирует в журнале аудита факт отправки и квитанцию оператора.
  7. Контрагент подписывает встречной КЭП. Оператор ЭДО возвращает квитанцию и подписанный экземпляр. Система проверяет подпись и обновляет статус документа на «Договор действует».

Результат

Договор создан, проверен, утверждён и подписан обеими сторонами. Все версии, визы, подписи и факты отправки зафиксированы в журнале аудита с криптографической защитой.

Исключения (нештатные ситуации)

№СценарийРеакция системы

1.2Конфликт версий при параллельном редактировании (два сотрудника правят один документ)Блокировка операции вторым редактором, предложение скачать актуальную версию, объединить и загрузить новую.

    → Документ переводится в статус «Ожидает подписания», Сервис уведомлений отправляет напоминание Администратору. При восстановлении – автоматическое подписание и отправка.

  • Справочник оргструктуры / HRIS не отвечает
    → Маршрутизатор использует кэшированную структуру (последний успешный запрос). В журнале аудита фиксируется пометка «Использован кэш». При длительной недоступности задача направляется Администратору для ручного назначения согласующего.

  • Модуль единого входа недоступен
    → Мобильное приложение / веб‑версия использует кэшированный токен, если сессия активна. При истечении токена – вход невозможен, отображается сообщение «Сервис аутентификации временно недоступен».

2.4. Сценарий 2. Служебная записка на оплату (Бухгалтер, бюджетный контроль)

Цель

Провести служебную записку / заявку на оплату счёта через Бухгалтера, с автоматической проверкой бюджетных лимитов (интеграция с ERP) и корректности реквизитов, с контролем версий и согласованием у Согласующего.

Условия начала

  • Инициатор авторизован в Dockies.
  • Справочник оргструктуры актуален – известен Согласующий (руководитель по иерархии) и Бухгалтер, ответственный за статью бюджета.
  • Интеграция с ERP / бюджетной системой активна (для проверки остатков).
  • Карточка документа содержит: тип «Служебная записка на оплату», сумму, код ЦФО, счёт на оплату, приложенный файл счёта.

Основной поток

  1. Инициатор создаёт служебную записку, заполняет карточку: контрагент, сумма, код ЦФО, прикрепляет файл счёта. Маршрутизатор определяет маршрут: первый шаг – Бухгалтер (проверка реквизитов и бюджетного лимита).
  2. Бухгалтер получает уведомление, открывает задачу:
    • Модуль интеграции с ERP автоматически проверяет реквизиты контрагента и остаток бюджета по коду ЦФО.
    • Если всё корректно – Бухгалтер нажимает «Согласовано по бюджету», система фиксирует визу.
  3. Маршрутизатор передаёт задачу Согласующему (руководителю Инициатора по иерархии HRIS).
  4. Согласующий просматривает контекст (реквизиты, сумму, остаток бюджета, визу Бухгалтера), накладывает УНЭП или КЭП (в зависимости от суммы/политики безопасности), система фиксирует подпись в журнале аудита.
  5. После подписи Маршрутизатор проверяет:
    • Если требуется дополнительное согласование (например, при сумме выше лимита руководителя) – передаёт вышестоящему Согласующему.
    • Если все согласования пройдены – обновляет статус служебной записки на «Оплатить» и формирует заявку на оплату в ERP (через интеграцию) или выгружает задачу Бухгалтеру для ручного запуска платежа.
  6. Инициатор и Бухгалтер получают уведомление о готовности документа. Спонсор проекта (при необходимости) – сводку по суммам заявок.

Результат

Служебная записка проверена по бюджетным лимитам и реквизитам, утверждена руководителем, зафиксирована в журнале аудита, передана в ERP для исполнения платежа.

Исключения

№СценарийРеакция системы
2.1Ошибка в реквизитах контрагента (ИНН/КПП/р/с)Бухгалтер получает задачу с пометкой «Реквизиты некорректны», вносит правки или отправляет на доработку Инициатору.
2.2Превышение бюджетного лимита по коду ЦФОЗадача блокируется, Бухгалтер не может согласовать, система предлагает выбрать другой ЦФО или запросить дополнительный бюджет.
2.3Смена цен/лимитов в процессе (обновилась ERP)Dockies фиксирует цены на момент отправки служебной записки и не пересчитывает задним числом. В случае расхождения – задача направляется Администратору.
2.4Отсутствие интеграции с ERP (сеть недоступна)Модуль проверки лимитов переходит в ручной режим – Бухгалтер получает задачу с пометкой «Проверить лимиты вручную», активирует элемент «Лимит подтверждён вручную» (берёт ответственность на себя), система фиксирует факт в журнале аудита.

Режимы деградации

  • ERP / бюджетная система недоступна
    → Автоматический переход в ручной режим (п. 2.4 выше). При восстановлении связи – синхронизация и сверка ручных операций.

  • Сервис уведомлений недоступен
    → Push-уведомления и email не отправлены. Dockies повторяет отправку через альтернативный канал (SMS → email), но не блокирует движение задачи (пользователь может увидеть задачу, зайдя в систему).

  • Модуль единого входа недоступен
    → Веб‑версия использует кэшированный токен активной сессии. При истечении токена – вход невозможен, на мобильных устройствах – аналогично.

2.5. Сценарий 3. Регистрация входящей корреспонденции и контроль поручений (Секретариат)

Цель

Зарегистрировать входящее письмо (бумажное или электронное), назначить ответственного (Согласующего), установить срок исполнения, контролировать выполнение поручения, зафиксировать результат в журнале аудита.

Условия начала

  • Секретарь авторизован в Dockies.
  • Входящий документ поступил на бумажном носителе или по электронной почте.
  • В системе настроен справочник типов входящих документов и шаблонов поручений.

Основной поток

  1. Секретарь создаёт карточку входящего документа, сканирует бумажный оригинал или прикрепляет электронный файл, заполняет реквизиты: отправитель, дата поступления, краткое содержание.
  2. Секретарь назначает Согласующего (руководителя, ответственного за исполнение) и устанавливает срок исполнения.
  3. Система генерирует поручение и отправляет уведомление Согласующему через Сервис уведомлений.
  4. Согласующий получает задачу, просматривает документ, выносит резолюцию (текстовый комментарий) и/или делегирует исполнение другому сотруднику.
  5. Согласующий отмечает поручение как исполненное (с приложением результата – файла ответа, отчёта).
  6. Система фиксирует все действия в журнале аудита (регистрация, назначение, резолюция, исполнение).
  7. Секретарь видит дашборд контроля поручений (статус, отклонение от срока). При необходимости может направить напоминание.

Результат

Входящий документ зарегистрирован, поручение назначено и выполнено в срок, все этапы зафиксированы в аудите. Обеспечена прозрачность контроля поручений.

Исключения

№СценарийРеакция системы
3.1Назначен несуществующий или уволенный сотрудникСистема блокирует назначение, предлагает актуальный список из HRIS.
3.2Пропущен срок исполнения порученияЕжедневное автоматическое уведомление Согласующему и Секретарю до закрытия поручения.
3.3Потеря сессии Секретаря при регистрацииDockies сохраняет черновик карточки, при повторном входе предлагает продолжить.

Режимы деградации

  • Справочник оргструктуры / HRIS недоступен
    → Секретарь может вручную ввести ФИО сотрудника (но система пометит как «не верифицировано»). При восстановлении связи – проверка и автозамена на актуальный ID.

  • Сервис уведомлений недоступен
    → Поручение создаётся, но уведомление не отправлено. Секретарь информирует сотрудника по телефону или email вручную. Система повторяет попытку отправки каждые 15 минут.

2.6. Сценарий 4. Кадровая служебная записка на отпуск/премию (Отдел кадров)

Цель

Сотрудник (Инициатор) подаёт служебную записку на отпуск или премирование. Кадровый специалист проверяет право на отпуск/премию (использованные дни, стаж, показатели), Согласующий (руководитель) утверждает, Бухгалтер (при премии) проводит начисление. Все версии и решения сохраняются в личном деле сотрудника.

Условия начала

  • Инициатор авторизован в Dockies.
  • Кадровый специалист (Отдел кадров) имеет доступ к личным делам и кадровым справочникам.
  • Для премии – интеграция с ERP для проверки бюджетных лимитов по фонду оплаты труда.

Основной поток

  1. Инициатор создаёт служебную записку типа «На отпуск» или «На премию», указывает период (для отпуска) / сумму и основание (для премии).
  2. Маршрутизатор направляет задачу в Отдел кадров.
  3. Кадровый специалист проверяет:
    • Для отпуска: остаток дней по личному делу, соблюдение графика отпусков.
    • Для премии: наличие оснований (положение о премировании, KPI).
    • При несоответствии – возвращает на доработку Инициатору с комментарием.
  4. После успешной проверки кадровый специалист нажимает «Согласовано по кадрам», система передаёт задачу Согласующему (руководителю Инициатора).
  5. Согласующий утверждает или отклоняет служебную записку.
  6. Если документ утверждён и это премия, Маршрутизатор направляет задачу Бухгалтеру для расчёта и включения в зарплатную ведомость (через ERP).
  7. Бухгалтер подтверждает начисление. Система фиксирует событие в журнале аудита и обновляет личное дело сотрудника (использованные дни отпуска, суммы премий).
  8. Инициатор получает уведомление о результате.

Результат

Служебная записка на отпуск или премию проверена кадрами, утверждена руководителем, при премии – передана в бухгалтерию. Изменения отражены в личном деле сотрудника.

Исключения

№СценарийРеакция системы
4.1У сотрудника недостаточно дней отпускаКадровый специалист отклоняет заявку с комментарием, система уведомляет Инициатора и предлагает перенести период.
4.2Для премии превышен бюджет ФОТБухгалтер получает задачу с пометкой «Бюджет ФОТ превышен», заявка возвращается Согласующему для дополнительного согласования с финансистами.
4.3Конфликт периодов отпуска с другим сотрудником в одном отделеКадровый специалист видит график отпусков, предлагает согласовать перенос.

Режимы деградации

  • Интеграция с ERP для бюджета ФОТ недоступна
    → Бухгалтер переходит в ручной режим: проверяет лимиты в альтернативной системе (таблица Excel) и фиксирует в Dockies пометку «Лимит проверен вручную». Запись в журнале аудита.

  • Личное дело сотрудника временно недоступно
    → Кадровый специалист запрашивает данные из внешней кадровой системы (резервный API). При полной недоступности – заявка ставится в очередь с пометкой «Ожидание данных», Инициатор уведомляется о задержке.

2.7. Сценарий 5. Выгрузка данных для проверки регулятором

Цель

Предоставить регулятору (налоговая инспекция, ФСБ/ФСТЭК, трудовая инспекция) выгрузку документов, подписей и аудиторских логов за требуемый период в соответствии с законодательством, с защитой от модификации и обеспечением конфиденциальности.

Условия начала

  • Поступил официальный запрос от регулятора (письменное требование).
  • Администратор системы авторизован с правами экспорта данных.
  • Включён режим аудита – все экспортные операции логируются.

Основной поток

  1. Администратор в специальном интерфейсе «Экспорт для регулятора» выбирает: тип документов (договоры, служебные записки, кадровые приказы и т.д.), диапазон дат, формат выгрузки (XML, JSON, структурированный PDF/A).
  2. Система формирует пакет выгрузки, включающий:
    • Карточки документов (все версии).
    • Электронные подписи (КЭП, УНЭП) с сертификатами и штампами времени.
    • Журнал аудита по каждому документу (кто, когда, что сделал, контрольные суммы).
  3. Система подписывает пакет квалифицированной электронной подписью Администратора (или технической учётной записи) и помещает его в защищённый архив.
  4. Администратор передаёт архив регулятору на внешнем носителе или через защищённый канал (портал госуслуг, ВКС).
  5. Служба безопасности получает уведомление о факте выгрузки, проверяет легитимность запроса и может наложить вето (отменить экспорт, если запрос недействителен). В случае вето – экспорт блокируется, событие фиксируется в журнале.
  6. Регулятор принимает пакет, при необходимости проверяет подпись и целостность данных.

Результат

Сформирована юридически значимая выгрузка, удовлетворяющая требованиям регулятора, с сохранением неотказуемости и неизменяемости. Все действия Администратора и Службы безопасности зафиксированы в аудите.

Исключения

№СценарийРеакция системы
5.1Запрашиваемый период содержит документы с грифом «Особая важность»Экспорт блокируется, требуется дополнительное разрешение от Службы безопасности.
5.2Некорректный запрос (нет основания, неправильный формат)Администратор возвращает запрос регулятору на доработку до начала выгрузки.
5.3Нет свободного места для формирования архиваСистема уведомляет Администратора, прерывает экспорт. После освобождения места – возможность продолжить.

Режимы деградации

  • Журнал аудита частично недоступен (сбой БД)
    → Экспорт разрешён только после восстановления целостности журнала. Администратор не может выгрузить неполные данные – система блокирует до исправления.

  • Криптопровайдер для подписи пакета недоступен
    → Выгрузка не может быть подписана КЭП. Система ожидает восстановления, уведомляет Администратора. Возможен ручной режим: подпись на внешнем носителе (USB-токен), но это требует отдельной процедуры.

3.1. Stakeholder requirements (SR‑SH)

SR‑SH‑1 (Инициатор — прозрачность и скорость)

1.1. Система должна позволять Инициатору создавать документ (заявку, договор, служебную записку), загружать файл, заполнять реквизиты и запускать его по маршруту согласования так, чтобы Инициатор в любой момент видел актуальный статус прохождения и получал уведомления о ключевых событиях (согласование, отклонение, наложение подписи, делегирование, эскалация, получение встречной подписи, финальный статус).

Верификация: демонстрация + инспекция интерфейса.

SR‑SH‑2 (Согласующий — удобство принятия решений)

2.1. Система должна предоставлять Согласующему задачу с полной информацией (документ, история версий, комментарии предыдущих участников, приложенные файлы), чтобы он мог принять решение (согласовать, отклонить, делегировать, отправить на доработку) без обращения к сторонним системам.

Верификация: демонстрация + функциональный тест.

SR‑SH‑3 (Юрист — контроль версий и правки)

3.1. Система должна обеспечивать Юристу возможность загружать отредактированную версию документа с сохранением предыдущих версий, а также просматривать различия между версиями, чтобы все участники маршрута видели, что именно было изменено.

Верификация: демонстрация + функциональный тест.

SR‑SH‑4 (Бухгалтер — проверка лимитов и реквизитов)

4.1. Система должна предоставлять Бухгалтеру при обработке заявки на оплату автоматически проверенные бюджетные лимиты (из интеграции с ERP) и статус реквизитов контрагента, а при недоступности ERP — позволять вручную подтвердить лимиты с фиксацией ответственности в аудите.

Верификация: демонстрация + интеграционный тест.

SR‑SH‑5 (Администратор — управляемость и мониторинг)

5.1. Система должна позволять Администратору настраивать маршруты согласования (последовательные, параллельные, с условиями), шаблоны документов, права доступа (роли → группы AD → типы документов) и просматривать журнал аудита с возможностью фильтрации по документу, пользователю, временному интервалу.

Верификация: демонстрация + функциональный тест.

SR‑SH‑6 (Клиент/Контрагент — юридически значимый обмен)

6.1. Система должна обеспечивать обмен документами с Контрагентом через оператора электронного документооборота с поддержкой встречной квалифицированной электронной подписи, чтобы обе стороны получали юридически значимые экземпляры, а также поддерживать протокол разногласий.

Верификация: интеграционный тест с тестовым оператором ЭДО.

SR‑SH‑7 (Служба безопасности — неотказуемость и аудит)

7.1. Система должна формировать неизменяемый криптографически связанный журнал аудита по каждому документу, фиксируя все действия (создание, загрузка версий, визы, подписи, отправка контрагенту). Журнал должен быть доступен для выгрузки и анализа Службе безопасности.

Верификация: инспекция архитектуры + тест целостности журнала.

SR‑SH‑8 (Спонсор проекта — отчётность по процессам)

8.1. Система должна предоставлять сводную отчётность по ключевым показателям: среднее время согласования документа, количество просроченных задач, загрузка по ролям, объём документооборота, чтобы Спонсор проекта мог оценить снижение операционных издержек и прозрачность процессов.

Верификация: демонстрация отчётов.

SR‑SH‑9 (Отдел кадров — автоматизация кадровых записок)

9.1. Система должна поддерживать автоматизированную обработку служебных записок на отпуск и премирование: проверку остатка дней отпуска из личного дела, передачу на согласование руководителю, начисление премии через интеграцию с ERP и обновление личного дела сотрудника.

Верификация: функциональный тест кадрового сценария.

SR‑SH‑10 (Секретариат — регистрация и контроль поручений)

10.1. Система должна позволять Секретарю регистрировать входящую/исходящую корреспонденцию (сканировать бумажные копии, прикреплять электронные файлы), назначать ответственных (Согласующих), устанавливать сроки исполнения, контролировать выполнение поручений и автоматически напоминать о просрочках.

Верификация: демонстрация + функциональный тест.

SR‑SH‑11 (Проектная команда — стабильность API)

11.1. Система должна предоставлять документированный API (REST или подобный) для всех ключевых операций (создание документа, получение статуса, подписание, выгрузка аудита) с версионированием, чтобы проектная команда могла разрабатывать интеграции и автотесты.

Верификация: инспекция документации API + тесты на соответствие.

SR‑SH‑12 (Регуляторы — соответствие законодательству)

12.1. Система должна обеспечивать возможность выгрузки полного пакета данных (документы, подписи, журнал аудита) по запросу регулятора (налоговая инспекция, ФСБ/ФСТЭК, трудовая инспекция) в читаемом формате (XML, JSON или PDF/A) с проверяемой целостностью, чтобы удовлетворять требованиям 63‑ФЗ, 125‑ФЗ и иных нормативных актов.

Верификация: инспекция архитектуры + тест экспорта.

SR‑SH‑13 (Служба безопасности — маскирование данных при экспорте)

13.1. При выгрузке документа для передачи внешнему лицу (контрагенту, регулятору) система должна предоставлять возможность маскирования конфиденциальных данных (суммы, персональные данные, коммерческая тайна) по настроенным Администратором правилам, с сохранением неизменяемого аудита факта маскирования.

Верификация: функциональный тест (маскирование → проверка выгрузки и аудита).

SR‑SH‑14 (Секретариат — контроль реестра поручений с календарём)

14.1. Система должна предоставлять Секретарю календарь поручений с визуальным отображением сроков, автоматическим напоминанием за 1 день до истечения и возможностью переназначения ответственного с уведомлением обеих сторон.

Верификация: демонстрация + функциональный тест.

SR‑SH‑15 (Отдел кадров — интеграция с графиком отпусков)

15.1. Система должна при создании служебной записки на отпуск автоматически проверять пересечение с уже утверждёнными отпусками других сотрудников того же отдела и при конфликте предлагать альтернативные даты или отправлять запрос руководителю на разрешение.

Верификация: интеграционный тест.

3.2. System requirements (SR‑SYS)

Формат: SR‑SYS‑X — системные требования (проверяемые, атомарные). Сгруппированы по функциональным областям.

Аутентификация и авторизация (сценарии 1–5)

SR‑SYS‑1 (Аутентификация через AD/SSO) 1.1. Модуль единого входа должен при входе пользователя отправлять запрос на проверку учётных данных в корпоративную службу каталогов (AD) и при успешном ответе выдавать токен сессии, содержащий: идентификатор пользователя, список ролей, список AD‑групп, email, атрибут manager (прямой руководитель), время истечения. 1.2. Верификация: интеграционный тест (вход → проверка полей токена).

SR‑SYS‑2 (Деградация аутентификации) 2.1. При недоступности службы каталогов система должна:

  • продолжать обслуживать пользователей с действующим токеном (до его истечения);
  • предоставлять Администратору интерфейс ручной аутентификации с фиксацией в журнале аудита записи «Резервная аутентификация» (ФИО администратора, время, обоснование). 2.2. Верификация: функциональный тест (отключение AD → проверка обоих сценариев).

SR‑SYS‑3 (Управление правами доступа) 3.1. Система должна предоставлять Администратору интерфейс управления ролевым доступом: создание ролей, назначение разрешений (создание документа, согласование, подписание КЭП/УНЭП, просмотр аудита, настройка маршрутов, управление справочниками), привязка ролей к пользователям (по логину AD). 3.2. Верификация: функциональный тест (создание роли → назначение → проверка доступа).

Создание и маршрутизация документов (сценарий 1)

SR‑SYS‑4 (Создание карточки документа) 4.1. Система должна предоставлять Инициатору возможность создать карточку документа с атрибутами: тип документа (договор / служебная записка / заявка), контрагент (если применимо), сумма, предмет, статья расходов, загруженный файл (Версия 1). Карточке присваивается уникальный идентификатор (UUID) и временная метка создания (UTC). 4.2. Верификация: функциональный тест (создание → проверка полей в БД).

SR‑SYS‑5 (Определение маршрута по справочнику оргструктуры) 5.1. После создания карточки Маршрутизатор должен обратиться к Справочнику оргструктуры (HRIS), определить последовательность согласования по типу документа и настроенному шаблону маршрута (последовательный, параллельный, с условиями) и сформировать первую задачу на указанную роль (юрист, бухгалтер, руководитель и т.д.). 5.2. Верификация: функциональный тест (создание договора → проверка: задача создана на роль «Юрист»).

SR‑SYS‑6 (Уведомление о новой задаче) 6.1. При формировании задачи на согласование система должна отправить уведомление адресату через Сервис уведомлений (email и/или push) с указанием: ID документа, тип, имя инициатора, крайний срок выполнения (если задан), прямая ссылка на задачу. 6.2. Верификация: интеграционный тест (задача создана → проверка доставки и содержимого уведомления).

SR‑SYS‑7 (Загрузка новой версии) 7.1. Участник маршрута (Юрист, Согласующий) должен иметь возможность загрузить новую версию файла с текстовым комментарием. Репозиторий документов должен сохранить файл, его контрольную сумму (SHA‑256), номер версии (автоинкремент) и связь с карточкой. 7.2. Верификация: функциональный тест (загрузка версии → проверка: файл, хеш, версия, связь).

SR‑SYS‑8 (Просмотр разницы версий) 8.1. Система должна предоставлять участнику маршрута возможность просмотреть список всех версий документа с атрибутами: номер версии, автор, дата, комментарий, контрольная сумма, и скачать любую из них. 8.2. Верификация: функциональный тест (создать 2 версии → проверка отображения списка и возможности скачать каждую).

SR‑SYS‑9 (Согласование и переход по маршруту) 9.1. При нажатии кнопки «Согласовано» система должна зафиксировать визу (ID пользователя, временная метка, решение) в журнале аудита и автоматически передать задачу следующему участнику маршрута согласно определённой последовательности. 9.2. Верификация: функциональный тест (виза Юриста → задача у Согласующего).

SR‑SYS‑10 (Отклонение и возврат на доработку) 10.1. При нажатии «Отклонить» система должна потребовать текстовый комментарий с причиной, зафиксировать отказ в журнале аудита, перевести документ в статус «Доработка» и вернуть задачу Инициатору. 10.2. Верификация: функциональный тест (отклонение → статус «Доработка» → задача у Инициатора).

SR‑SYS‑11 (Обнаружение конфликта версий) 11.1. Если с момента открытия задачи текущим участником другой участник загрузил новую версию файла, при попытке текущего участника согласовать или загрузить свою версию система должна отобразить предупреждение о конфликте и заблокировать операцию до разрешения конфликта (принять новую версию, загрузить поверх, эскалировать инициатору). 11.2. Верификация: функциональный тест (параллельная загрузка двух версий → предупреждение, блокировка).

Электронная подпись (сценарии 1, 2)

SR‑SYS‑12 (Наложение КЭП) 12.1. При подписании документа КЭП Модуль электронной подписи должен:

  • вычислить SHA‑256 хеш подписываемого файла;
  • отправить хеш в криптопровайдер (УЦ);
  • получить подписанный хеш и штамп времени (RFC 3161);
  • сохранить подпись в карточке документа (формат PKCS#7);
  • записать в журнал аудита: ID подписанта, хеш, серийный номер сертификата, временную метку, результат проверки OCSP. 12.2. Верификация: интеграционный тест (подписание → проверка сохранённой подписи, аудита).

SR‑SYS‑13 (Проверка сертификата перед подписанием) 13.1. Перед наложением подписи система должна проверить статус сертификата через протокол OCSP. Если сертификат отозван, истёк или недействителен — заблокировать кнопку «Подписать» и предложить пользователю перевыпустить сертификат или делегировать подписание. 13.2. Верификация: функциональный тест (отозванный сертификат → кнопка заблокирована).

SR‑SYS‑14 (Деградация подписи) 14.1. При недоступности криптопровайдера система должна перевести документ в статус «Ожидает подписания (офлайн)», разрешить проставление внутренней визы «Подписано вручную» без юридической силы (с пометкой в аудите) и поставить документ в очередь на автоматическое подписание при восстановлении связи с криптопровайдером. 14.2. Верификация: функциональный тест (отключение криптопровайдера → статус, виза, очередь).

Обмен с контрагентом через оператора ЭДО (сценарий 1)

SR‑SYS‑15 (Отправка документа контрагенту) 15.1. После подписания КЭП всеми внутренними участниками Модуль интеграции с ЭДО должен отправить документ контрагенту через внешнего оператора (формат XML, подписанный пакет) и зафиксировать в журнале аудита факт отправки, ID квитанции оператора, временную метку. 15.2. Верификация: интеграционный тест (подписание → отправка → квитанция в аудите).

SR‑SYS‑16 (Приём встречной подписи и протокола разногласий) 16.1. При получении от оператора ЭДО подписанного контрагентом документа система должна проверить КЭП контрагента (через OCSP), сохранить экземпляр в репозитории, уведомить Инициатора и обновить статус документа на «Договор действует». При получении протокола разногласий — вернуть задачу Инициатору/Юристу с комментариями контрагента. 16.2. Верификация: интеграционный тест (входящий документ → проверка подписи → уведомление, смена статуса).

Мобильное согласование (сценарий 2)

SR‑SYS‑17 (Согласование с мобильного устройства) 17.1. Система должна предоставлять мобильный интерфейс (адаптивная веб-версия или приложение), позволяющий Согласующему:

  • просматривать документ (PDF);
  • просматривать историю версий;
  • выбирать решение (согласовать / отклонить / делегировать);
  • подписывать УНЭП через код подтверждения, отправленный по SMS или push. 17.2. Верификация: функциональный тест на мобильном устройстве (Android/iOS).

SR‑SYS‑18 (Оффлайн‑режим мобильного приложения) 18.1. При отсутствии сетевого соединения мобильное приложение должно сохранять решение пользователя локально (в зашифрованном хранилище) и синхронизировать с сервером при восстановлении связи, добавляя в аудит пометку «Подписано в оффлайн‑режиме» и фактическое время получения решения на сервере. 18.2. Верификация: функциональный тест (оффлайн → решение → восстановление → проверка аудита).

SR‑SYS‑19 (Отложенная подпись при отсутствии сертификата) 19.1. Если документ требует КЭП, а на мобильном устройстве сертификат отсутствует, система должна перевести задачу в статус «Ожидает подпись с ПК» с указанием срока (по умолчанию 24 часа) и отправить напоминание пользователю (email / push). 19.2. Верификация: функциональный тест (мобильное устройство → нет сертификата → статус, напоминание).

Заявка на оплату и контроль лимитов (сценарии 2, 4)

SR‑SYS‑20 (Автоматическая проверка лимитов через ERP) 20.1. При создании заявки на оплату система должна автоматически запросить у ERP: остаток бюджета по статье расходов, лимит договора, статус контрагента. Затем применить политику:

  • если остаток ≥ суммы заявки и контрагент активен — направить задачу Бухгалтеру;
  • если контрагент активен, но остаток < суммы — передать на эскалацию (SR‑SYS‑21);
  • если контрагент неактивен или не найден — заблокировать отправку с уведомлением Инициатору. 20.2. Верификация: функциональный тест (три варианта → проверка реакции).

SR‑SYS‑21 (Эскалация при превышении лимита) 21.1. При превышении бюджетного лимита Маршрутизатор должен найти через Справочник оргструктуры вышестоящего руководителя (финансового директора) и добавить его в цепочку согласования. Заявка переходит в статус «Ожидание бюджета». При одобрении → задача Бухгалтеру. При отклонении → возврат Инициатору с комментарием. 21.2. Верификация: функциональный тест (превышение → задача у финдиректора → одобрение/отклонение).

SR‑SYS‑22 (Деградация при недоступности ERP) 22.1. При недоступности ERP система должна:

  • отобразить Бухгалтеру пометку «Проверить лимиты вручную»;
  • предоставить чекбокс «Лимит подтверждён вручную»;
  • при активации чекбокса зафиксировать в аудите: ID бухгалтера, временную метку, пометку «Ручной ввод, ERP недоступна». 22.2. Верификация: функциональный тест (отключение ERP → появление пометки, чекбокса, запись аудита).

SR‑SYS‑23 (Блокировка при отсутствии контрагента) 23.1. Если ERP возвращает «контрагент не найден», система должна заблокировать отправку заявки Бухгалтеру, уведомить Инициатора с текстом «Контрагент отсутствует в ERP. Заведите карточку контрагента и повторите заявку». 23.2. Верификация: функциональный тест (контрагент отсутствует → блокировка, уведомление).

Регистрация входящей корреспонденции (сценарий 3)

SR‑SYS‑24 (Создание карточки входящего документа) 24.1. Секретарь должен иметь возможность создать карточку входящего документа с атрибутами: входящий номер, дата поступления, отправитель, краткое содержание, сканированный файл или прикреплённый электронный документ. Система присваивает уникальный ID карточки. 24.2. Верификация: функциональный тест (регистрация → проверка атрибутов в БД).

SR‑SYS‑25 (Назначение поручения и контроль срока) 25.1. Секретарь может назначить ответственного (Согласующего) из справочника оргструктуры, установить срок исполнения. Система должна создать задачу для Согласующего, отправить уведомление и вести контроль срока (ежедневные напоминания за 2 дня до истечения, эскалация при просрочке). 25.2. Верификация: функциональный тест (назначение → проверка задачи, напоминаний).

Кадровая служебная записка (сценарий 4)

SR‑SYS‑26 (Проверка остатка дней отпуска) 26.1. При создании служебной записки на отпуск система должна запросить у кадрового модуля (интеграция с HRIS) остаток дней отпуска сотрудника. Если остаток меньше запрашиваемого — заблокировать отправку и уведомить Инициатора с указанием доступных дней. 26.2. Верификация: интеграционный тест (заявка на 10 дней при остатке 5 → блокировка).

SR‑SYS‑27 (Автоматическое обновление личного дела после отпуска/премии) 27.1. После утверждения служебной записки на отпуск или премию система должна передать в HRIS данные: тип события, период отпуска (или сумма премии), дату утверждения. HRIS обновляет личное дело сотрудника. 27.2. Верификация: интеграционный тест (утверждение отпуска → проверка данных в HRIS).

Выгрузка для регулятора (сценарий 5)

SR‑SYS‑28 (Формирование пакета выгрузки) 28.1. Администратор должен иметь интерфейс для выгрузки данных по запросу регулятора: выбор типов документов, диапазона дат, формата (XML, JSON, PDF/A). Система должна сформировать архив, включающий:

  • карточки документов (все версии);
  • подписи (КЭП, УНЭП) с сертификатами и штампами времени;
  • полный журнал аудита по каждому документу (лента событий, контрольные суммы). 28.2. Верификация: функциональный тест (выгрузка за период → проверка состава архива).

SR‑SYS‑29 (Криптографическая подпись пакета выгрузки) 29.1. Сформированный пакет должен быть подписан КЭП технической учётной записи Dockies (или Администратора), чтобы гарантировать целостность и подлинность выгруженных данных. Подпись сохраняется вместе с архивом. 29.2. Верификация: интеграционный тест (выгрузка → проверка подписи внешним средством).

Аудит и архивация (все сценарии)

SR‑SYS‑30 (Неизменяемый журнал аудита) 30.1. Каждое значимое действие (создание, загрузка версии, виза, подпись, отправка через ЭДО, изменение статуса, экспорт) должно фиксироваться в Журнале аудита с атрибутами: ID события, ID пользователя, временная метка (UTC с точностью до миллисекунды), IP‑адрес устройства, тип действия, ID документа, контрольная сумма файла (если применимо). Записи связываются криптографической цепочкой: каждая запись содержит хеш предыдущей. 30.2. Верификация: инспекция архитектуры + функциональный тест (серия действий → проверка цепочки хешей).

SR‑SYS‑31 (Экспорт акта аудита) 31.1. Система должна предоставлять возможность экспортировать акт аудита по конкретному документу в формате PDF/A, содержащий: полную ленту событий, контрольные суммы файлов, данные подписей (серийные номера сертификатов, штампы времени). Акт должен содержать системную печать платформы (метка времени, идентификатор экземпляра). 31.2. Верификация: функциональный тест (экспорт → проверка содержимого PDF).

SR‑SYS‑32 (Передача в долговременный архив) 32.1. После завершения всех согласований и подписаний (включая встречную КЭП контрагента, если применимо) система должна передать завершённое дело в Долговременный архив (ECM) в течение 24 часов. Передаваемый пакет включает: файлы всех версий, подписи, метаданные карточки, ленту аудита. Статус документа меняется на «Архивирован». Система должна получить от архива квитанцию с идентификатором дела и контрольной суммой. 32.2. Верификация: интеграционный тест (завершение → проверка передачи, квитанции, статуса).

Справочники и синхронизация

SR‑SYS‑33 (Синхронизация справочника оргструктуры с HRIS) 33.1. Справочник оргструктуры должен синхронизироваться с HRIS с настраиваемым интервалом (по умолчанию 1 раз в сутки) и обновлять данные: сотрудники, роли, иерархия подчинения (атрибут manager), временные замещения. 33.2. Верификация: интеграционный тест (изменение в HRIS → синхронизация → проверка справочника).

SR‑SYS‑34 (Деградация при недоступности HRIS) 34.1. При недоступности HRIS Маршрутизатор должен использовать последнюю кэшированную версию справочника (с пометкой в аудите «Использован кэш, HRIS недоступна»). Задачи назначаются по ролям (не по конкретным именам). При восстановлении — автоматическая синхронизация. 34.2. Верификация: функциональный тест (отключение HRIS → маршрутизация по кэшу → запись аудита).

SR‑SYS‑35 (Кэширование справочника контрагентов) 35.1. Для ускорения проверки реквизитов контрагентов при создании заявок система должна кэшировать базовые данные (ИНН, КПП, наименование) из ERP с обновлением не реже 1 раза в сутки. При недоступности ERP используется кэш (с пометкой в аудите). 35.2. Верификация: функциональный тест (отключение ERP → создание заявки с существующим контрагентом → использование кэша).

Уникальные требования для Dockies

SR‑SYS‑36 (Настройка правил маскирования) 36.1. Администратор должен иметь интерфейс настройки правил маскирования по типам документов и ролям получателя: например, для роли «Контрагент» скрывать сумму НДС, для роли «Регулятор» — скрывать персональные данные физических лиц. Маскирование применяется к выгружаемому PDF/XML, оригинал в системе остаётся неизменным. В аудит записывается: кто, когда, какой документ, какое правило маскирования применено. 36.2. Верификация: функциональный тест (настройка правила → экспорт с маскированием → проверка результата и аудита).

SR‑SYS‑37 (Виджет «Календарь поручений») 37.1. Система должна предоставлять Секретарю виджет «Календарь поручений» с возможностью фильтрации по статусу (выполнено/просрочено/в работе), цветовой индикацией сроков (зелёный — более 3 дней, жёлтый — 1–3 дня, красный — просрочено). При клике на поручение открывается карточка документа. 37.2. Верификация: демонстрация + функциональный тест (фильтры, цветовая индикация, переход).

SR‑SYS‑38 (Автоматическое напоминание о поручении) 38.1. За 1 календарный день до истечения срока исполнения поручения система должна отправить напоминание ответственному и Секретарю (email, push). При истечении срока — отправить уведомление руководителю ответственного (из HRIS) и Секретарю. 38.2. Верификация: функциональный тест (создать поручение со сроком завтра → проверка уведомлений за день и в день истечения).

SR‑SYS‑39 (Проверка пересечения отпусков) 39.1. При создании служебной записки на отпуск система должна запросить у кадрового модуля (HRIS) список уже утверждённых отпусков сотрудников того же отдела за указанный период. Если пересечение более чем на 50% периода — заблокировать отправку с сообщением «В отделе уже запланировано отсутствие [доля] сотрудников. Согласуйте перенос с руководителем». При меньшем пересечении — предупреждение без блокировки. 39.2. Верификация: интеграционный тест (создание отпуска при конфликте >50% → блокировка, сообщение).

SR‑SYS‑40 (Фиксация идентификатора устройства) 40.1. Система должна фиксировать в журнале аудита идентификатор устройства (fingerprint браузера или уникальный ID мобильного приложения) при каждой операции подписания и согласования. При обнаружении подписания с неуправляемого (личного) устройства, не внесённого в белый список Администратором, — отправлять уведомление Службе безопасности. 40.2. Верификация: функциональный тест (подписание с двух устройств → проверка аудита, уведомление).

SR‑SYS‑41 (Экспорт по требованиям Трудовой инспекции) 41.1. По требованию Трудовой инспекции система должна формировать выгрузку по кадровым документам (приказы о приёме/увольнении, отпуска) в формате, установленном нормативными актами (XML по XSD ФНС или аналоги), с возможностью передачи через защищённый канал (СМЭВ или портал госуслуг). Выгрузка подписывается КЭП организации. 41.2. Верификация: инспекция + интеграционный тест (генерация XML по XSD, проверка подписи).

Методика измерения SLA для проекта Dockies

Определения и правила:

  • Доступность (Availability) — доля времени, в течение которого критичные функции платформы (создание, согласование, подписание) принимают и корректно обрабатывают запросы. Измеряется как:

    Доступность (%) = ((Т_рабочее − Т_простоя) / Т_рабочее) × 100%

    где:

    • Т_рабочее — суммарное рабочее время за календарный месяц (понедельник–пятница, 08:00–20:00 по московскому времени);
    • Т_простоя — суммарное время, когда функция не отвечала или отвечала с ошибкой (HTTP 5xx, тайм-аут, некорректный ответ). Плановые окна обслуживания (согласованные с администратором и опубликованные не менее чем за 2 недели) исключаются из Т_простоя, но включаются в Т_рабочее только если выходят за согласованный лимит (4 часа в месяц).

  • Время отклика (Response Time) — интервал от момента отправки запроса пользователем (нажатие кнопки в веб-интерфейсе) до момента отображения результата (успех/ошибка). Измеряется на стороне сервера приложений (без учёта задержки сети клиента). Отчётные перцентили: p50, p95, p99.

  • Нагрузочный профиль — при указании «при нагрузке до N одновременных пользователей» подразумевается, что N пользователей одновременно выполняют типичные операции (создание документа, согласование, просмотр). Профиль моделируется при нагрузочном тестировании. Конкретная инфраструктура (количество серверов, CPU, RAM) определяется на этапе проектирования по результатам тестирования и не является предметом требований.

  • Целевая точка восстановления (RPO) — максимально допустимый объём потерянных данных, выраженный во времени. Значение 15 минут означает, что при аварии могут быть потеряны данные, введённые не более чем за последние 15 минут.

  • Целевое время восстановления (RTO) — максимально допустимое время от момента аварии до полного возобновления работы платформы.

  • Отчётный период — календарный месяц. Отчёт по SLA формируется автоматически на основе журналов платформы и включает: доступность по каждой критичной функции, распределение времени отклика (p50, p95, p99), количество и длительность инцидентов, использованное время плановых окон.

4.1. Декомпозиция на компоненты и обязанности

Архитектура платформы Dockies строится по принципу модульного монолита с чётким выделением доменных сервисов. Это позволяет сохранить высокую производительность и простоту развёртывания, присущие монолиту, но при этом обеспечить логическую изоляцию функциональных областей.

4.2. Ключевые архитектурные решения (ADR)

ADR-1: Логический монолит с доменными сервисами

Контекст:
Требования к системе включают высокую производительность (NFR‑1, NFR‑2) и поддержку до 500 одновременных пользователей (NFR‑7). Команда разработки небольшая (~5 человек). Необходимо быстро выводить новые функции.

Решение:
Реализовать систему как логический монолит, где компоненты разделены на уровне кода и вызовов, но деплоятся как единое приложение. Каждый доменный сервис (документы, маршрутизация, аудит) изолирован и общается через чёткие интерфейсы.

Отвергнутая альтернатива:
Микросервисная архитектура с независимым деплоем каждого компонента.

КритерийМодульный монолитМикросервисы
Сложность разработкиНизкая — единая кодовая база, простой отладчикВысокая — распределённая трассировка, управление конфигурацией
ПроизводительностьВысокая — вызовы внутри процессаНиже — сетевые вызовы, сериализация
Простота деплояОчень простая — один артефактСложная — оркестрация многих сервисов
Масштабирование частейНе поддерживается — масштабируется только весь монолитГибкое — можно масштабировать только нагруженные сервисы

Последствия:

  • Простота разработки и ускорение вывода функций на рынок.
  • Нулевые накладные расходы на межсервисное взаимодействие, что помогает соблюсти SLA по времени отклика (NFR‑1, NFR‑2).
  • Единая база данных с транзакционной целостностью (SR‑SYS‑30) без сложности распределённых транзакций.
  • При росте нагрузки масштабировать придётся всё приложение целиком (компромисс приемлем для целевых 500 пользователей).

ADR-2: Криптографическая цепочка в реляционной БД для аудита

Контекст:
Требование неизменяемости журнала аудита (SR‑SH‑7) при стандартном стеке технологий (PostgreSQL). Необходимо найти баланс между юридической значимостью (SR‑SH‑12) и стоимостью решения.

Решение:
Хранить журнал аудита в отдельной, защищённой схеме реляционной БД, связывая записи криптографической цепочкой. Каждая новая запись содержит хеш предыдущей (SHA‑256), что позволяет обнаружить факт модификации. Проверка целостности запускается ежечасно.

Отвергнутая альтернатива:
Использование WORM-носителей (однократная запись) или специализированных блокчейн-платформ.

КритерийКриптоцепочка в БДWORM-носитель
СтоимостьНулевая аппаратная — реализуется программноТребует специализированного оборудования
Гибкость развёртыванияРаботает на любой инфраструктуре, включая облачнуюПривязка к физическому носителю
Защита от модификацииОбнаруживает модификацию, но не предотвращает физическиФизически невозможна перезапись
Восстановление данныхНе восстанавливает повреждённые записиДанные не повреждаются

Последствия:

  • Отсутствие зависимости от дорогого оборудования. Платформа может разворачиваться в облаке.
  • Программная проверка целостности в автоматическом режиме.
  • Между модификацией записи и её обнаружением есть временной промежуток (до часа).
  • Обнаружение факта нарушения не восстанавливает данные — требуется резервная копия (инкрементальные копии каждые 4 часа).
  • Прямой доступ к БД журнала строго ограничен на уровне инфраструктуры.

ADR-3: Единая реляционная база для структурированных данных

Контекст:
Платформа хранит карточки документов, задачи, записи аудита, роли и шаблоны. Ожидаемый объём — до 1 000 000 карточек и 5 ТБ файлов в объектном хранилище, до 500 одновременных пользователей.

Решение:
Все структурированные данные (атрибуты, связи, статусы) хранить в единой реляционной БД (PostgreSQL). Журнал аудита — в отдельной схеме с ограниченными правами. Файлы документов — в объектном хранилище (S3). Тяжёлые аналитические запросы направлять на реплику для чтения.

Отвергнутая альтернатива:
Отдельная база для каждого логического модуля или нереляционное хранилище для аудита.

КритерийЕдиная реляционная БДРаздельные хранилища
Согласованность данныхТранзакционная целостность (ACID)Требуются распределённые транзакции
Сложность администрированияОдна база — единая политика бэкаповМножество политик, больше точек отказа
Запросы между модулямиПростые JOIN (например, документ + задача)Сложная агрегация на уровне приложения

Последствия:

  • Транзакционная целостность данных без распределённых транзакций (SR‑SYS‑30).
  • Низкая сложность администрирования и резервного копирования (NFR-8).
  • Простота и скорость запросов между модулями.
  • Потенциальное узкое место: высокая частота записи в аудит может конкурировать за ресурсы (смягчается через реплику).
  • При объёме 1 млн документов нагрузка укладывается в возможности одного сервера.

ADR-4: Асинхронная очередь для взаимодействия с внешними системами

Контекст:
Dockies интегрируется с AD, ERP, HRIS, криптопровайдером, оператором ЭДО и шлюзом уведомлений. Внешние системы могут быть недоступны (NFR-4).

Решение:
Все вызовы к внешним системам, результат которых не требуется пользователю немедленно, помещаются в асинхронную очередь (RabbitMQ) с повторными попытками (retry с exponential backoff). Синхронными остаются только вызовы, результат которых необходим для продолжения операции пользователя (проверка сертификата OCSP, проверка лимитов). Для синхронных вызовов установлен таймаут (3 секунды) с переходом в режим деградации.

Отвергнутая альтернатива:
Все вызовы к внешним системам — синхронные. При недоступности внешней системы операция пользователя блокируется.

КритерийАсинхронная очередьПолностью синхронные вызовы
Время отклика для пользователяБыстрый ответ — операция поставлена в очередьЗависит от времени ответа внешней системы
Поведение при сбое внешней системыОперация сохраняется в очереди, пользователь продолжает работуОперация блокируется или завершается ошибкой
Гарантия доставкиВысокая — очередь с повторами и сохранениемНизкая — при ошибке требуется ручной повтор
Сложность реализацииСредняя — требуется очередь, обработка повторов, идемпотентностьНизкая — простой запрос-ответ

Последствия:

  • Устойчивость к сбоям внешних систем — отправка уведомлений, документов и синхронизация данных не блокируют пользователя и выполняются при восстановлении (NFR-3).
  • Соблюдение таймингов отклика пользовательского интерфейса (NFR-1, NFR-2).
  • Немедленная синхронизация невозможна — информация может появляться с небольшой задержкой (секунды–минуты).
  • Отладка асинхронных цепочек сложнее, но компенсируется централизованным логированием.
  • Идемпотентность достигается уникальными идентификаторами операций.

ADR-5: Адаптивная веб-версия для мобильного доступа

Контекст:
Система должна поддерживать мобильный доступ (SR‑SH‑2, SR‑SH‑8), включая просмотр, согласование и подписание УНЭП по SMS-коду.

Решение:
Основной мобильный интерфейс реализовать как адаптивную веб-версию с PWA и поддержкой оффлайн-кэширования через сервис-воркер. Нативное приложение — опционально на следующем этапе.

Отвергнутая альтернатива:
Разработка нативных приложений для Android и iOS с первого этапа.

КритерийАдаптивная веб-версия (PWA)Нативные приложения

Последствия:

  • Быстрый выход на мобильную платформу с низкими затратами.
  • Мгновенные обновления интерфейса для всех пользователей.
  • Возможность согласования и подписания УНЭП через браузер (SR‑SYS‑17).
  • Полноценная работа без сети невозможна, подписание КЭП на мобильном устройстве не поддерживается (требуется ПК).
  • Сценарий, требующий подписания КЭП с мобильного устройства, обрабатывается через статус «Ожидает подпись с ПК» (SR‑SYS‑19).
  • При необходимости функциональность нативных приложений будет добавлена на следующих этапах.

6.1 Шкала оценки и методология

Для количественной оценки и приоритизации рисков используется двухфакторная модель.

Вероятность (P):

  • 1 – Низкая (<10% за отчётный период) – событие маловероятно в течение года.
  • 2 – Средняя (10–50%) – событие возможно в течение полугода.
  • 3 – Высокая (>50%) – событие ожидаемо в течение квартала.

Влияние (I):

  • 1 – Низкое – незначительные задержки (до 1 рабочего дня) или потери <100 тыс. руб.
  • 2 – Среднее – остановка отдельных процессов, финансовые потери 100–500 тыс. руб., нарушение SLA до 2 часов.
  • 3 – Высокое – остановка ключевых бизнес-процессов, финансовые потери >500 тыс. руб., крупные штрафы (≥1 млн руб.), юридическая ответственность, репутационный ущерб.

Приоритет = P × I (от 1 до 9).

  • Приоритет 6–9 – высокий (обязателен детальный план реагирования).
  • Приоритет 4 – средний (требуется наблюдение и базовые меры).
  • Приоритет 1–3 – низкий (принимается без дополнительных мер).

6.4 Детальные планы реагирования для рисков высокого приоритета

R‑01: Недоступность оператора ЭДО

Предотвращение:

  • Очередь RabbitMQ с экспоненциальным ретраем (1с,2с,4с,8с,16с), 5 попыток.
  • Ручной канал: выгрузка PDF для подписания по email + курьер (для критических документов).
  • Мониторинг статуса квитанции ЭДО: таймаут 30 сек → алерт.
  • Еженедельное тестирование интеграции.

Обнаружение:

  • Метрика "возраст самого старого сообщения в очереди ЭДО" >5 мин → алерт.
  • Health‑API оператора ЭДО каждые 5 мин.

Реагирование и восстановление:

  • После 3 неудачных попыток → уведомление Администратора, предложение ручной отправки.
  • Кнопка "Повторить отправку" с идемпотентностью.

Fallback / деградация:

  • Статус документа «Ожидает отправки (ЭДО недоступен)».
  • Контрагент уведомляется о задержке.

R‑04: Истечение или отзыв сертификата КЭП

Предотвращение:

  • Проверка через OCSP перед каждой операцией подписания (таймаут 3с).
  • Уведомления владельца за 30/7/1 день до истечения.
  • Блокировка кнопки подписи при недействительном сертификате.

Обнаружение:

  • Журнал всех попыток подписания.
  • Еженедельный отчёт администратору.
  • Алерт при попытке использования отозванного сертификата.

Реагирование:

  • Делегирование подписания другому сотруднику через МЧД.
  • Внутренняя виза с пометкой (временная мера).
  • Экстренный перевыпуск сертификата (SLA 4 часа).

R‑07: Выгрузка неполных или модифицированных данных для регулятора

Предотвращение:

  • Пакет подписывается КЭП с контрольной суммой (SHA‑256) архива.
  • Выборка через реплику БД.
  • Криптоцепочка аудита с обнаружением модификации ≤1 мин.

Обнаружение:

  • Автоматическая валидация пакета после формирования.
  • Логирование экспортных операций.

Реагирование:

  • Ручная верификация пакета перед отправкой.
  • При ошибке → автоматическая регенерация.
  • Резервное копирование выгрузок на 30 дней.

R‑08: DDoS‑атака на инфраструктуру

Предотвращение:

  • CDN с фильтрацией трафика.
  • WAF (OWASP Top 10).
  • Автомасштабирование (K8s HPA).
  • Регулярное нагрузочное тестирование.

Обнаружение:

  • Мониторинг аномального трафика, алерт в SOC.
  • Автоматическая активация капчи / проверки браузера.

Реагирование:

  • Переключение на резервную площадку.
  • Rate limiting для подозрительных IP.
  • Привлечение анти‑DDoS провайдера.

Деградация:

  • Отключение некритичных функций (отчёты, дашборды).
  • Ограничение доступа только авторизованным пользователям.

R‑09: Утечка конфиденциальных данных

Предотвращение:

  • Мандатный доступ (AD группы).
  • Маскирование данных при экспорте.
  • Шифрование файлов (AES‑256).
  • Обучение персонала, пентесты.

Обнаружение:

  • IPS/IDS, SIEM.
  • Мониторинг аномальных операций (массовое скачивание, ночной доступ).

Реагирование:

  • Блокировка учётной записи нарушителя, смена ключей шифрования.
  • Служебное расследование.
  • Уведомление регулятора в течение 24 ч (152‑ФЗ).

Деградация:

  • Отключение доступа к конфиденциальным документам.
  • Перевод платформы в режим «только чтение».

R‑14: Потеря данных из‑за отказа дисков

Предотвращение:

  • Ежедневный полный бекап + инкремент каждые 4 часа, хранение 30 дней.
  • Репликация (RAID 10, geo‑репликация S3).
  • Версионирование S3.

Обнаружение:

  • Мониторинг SMART-статуса дисков.
  • Логирование ошибок ввода-вывода.
  • Регулярное тестирование восстановления из бекапов.

Реагирование:

  • Автоматическое переключение на реплику.
  • Восстановление из последнего бекапа (RTO ≤1ч, RPO ≤15мин).
  • План аварийного восстановления (DRP).

TC-01: E2E создание и маршрутизация договора (Happy Path)

  • ID: TC-01
  • Сценарий: 1 (Создание и согласование договора)
  • Цель: Подтвердить, что инициатор может создать карточку договора, а система корректно определяет маршрут согласования и создаёт задачу первому участнику.
  • Предусловия:
    • Пользователь "Инициатор" авторизован в Dockies.
    • Справочник оргструктуры актуален, для типа документа "Договор" настроен маршрут: "Юрист" -> "Согласующий".
    • Шаблон договора доступен для загрузки.
  • Тестовые данные:
    • Тип документа: contract
    • Имя контрагента: "ООО ТестКонтрагент"
    • Сумма договора: 1 000 000 руб.
    • Файл: draft_contract.pdf
  • Шаги:
    1. Инициатор нажимает «Создать документ».
    2. Заполняет все обязательные поля карточки (тип, контрагент, сумма, предмет).
    3. Загружает файл draft_contract.pdf (Версия 1).
    4. Нажимает кнопку «Отправить на согласование».
    5. Проверяет, что в интерфейсе отображается новый статус документа.
  • Ожидаемый результат:
    • После шага 4, система возвращает 201 Created и присваивает документу уникальный UUID.
    • Сервис маршрутизации создаёт задачу на роль "Юрист".
    • Юрист получает уведомление о новой задаче.
    • В карточке документа отображается статус: in_progress и текущий этап: legal_review.
  • Постусловия: Созданная карточка и её версия сохранены в БД.

TC-02: Отклонение документа и возврат на доработку

  • ID: TC-02
  • Сценарий: 1 (Создание и согласование договора)
  • Цель: Проверить, что участник маршрута может отклонить документ, после чего он возвращается инициатору на доработку.
  • Предусловия: Документ находится на этапе согласования у пользователя "Согласующий".
  • Шаги:
    1. Пользователь "Согласующий" открывает свою задачу на согласование.
    2. Нажимает кнопку «Отклонить».
    3. Вводит комментарий с причиной отклонения: «Неверно указана сумма договора».
    4. Подтверждает действие.
  • Ожидаемый результат:
    • Статус документа меняется на rejected.
    • Задача на согласование у "Согласующего" закрывается.
    • Создаётся новая задача для пользователя "Инициатор" со статусом Доработка.
    • "Инициатор" получает уведомление об отклонении с указанной причиной.
  • Постусловия: В журнале аудита зафиксировано событие отклонения с комментарием.

  • Предусловия: На этапе согласования у пользователя "Согласующий" есть задача по договору. Юрист уже загрузил правки (Версия 1.1).

    3. Открывает вкладку «Версии».
