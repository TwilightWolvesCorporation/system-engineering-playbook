## 2.1. Участники (стейкхолдеры / роли)

| **Группа**              | **Роль в документообороте Dockies**                                                                                                                                  |
|-------------------------|----------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| **Инициатор**           | Сотрудник, создающий документ (заявка, договор, служебная записка), отслеживающий статус согласования и получающий результат.                                        |
| **Согласующий**         | Руководитель или менеджер, принимающий решение: утвердить, отклонить, делегировать или отправить на доработку.                                                       |
| **Юрист**               | Сотрудник юридического отдела, проверяющий документы (преимущественно договоры) на соответствие законодательству и внутренним нормативам.                            |
| **Бухгалтер**           | Сотрудник бухгалтерии, проверяющий реквизиты в договорах, связывающий заявки с бюджетом и инициирующий оплату.                                                       |
| **Администратор**       | Специалист IT или подразделения кадров, настраивающий маршруты согласования, права доступа (AD‑группы), справочники и шаблоны, а также контролирующий логи аудита.   |
| **Клиент / Контрагент** | Внешняя организация или физическое лицо – сторона по договору, получающая подписанные документы через оператора ЭДО.                                                 |
| **Служба безопасности** | Внутренний отдел, контролирующий доступ к конфиденциальной информации, неотказуемость действий, защиту от утечек и проверяющий логи аудита.                          |
| **Спонсор проекта**     | Акционер, получающий отчётность о времени согласования, операционных издержках и прозрачности процессов.                                                             |
| **Отдел кадров**        | Кадровый специалист, автоматизирующий служебные записки (отпуска, премии), предоставляющий актуальную оргструктуру и штатное расписание.                             |
| **Секретариат**         | Секретарь, регистрирующий входящую/исходящую корреспонденцию, контролирующий поручения и организующий подписание бумаг у руководства.                                |
| **Проектная команда**   | Разработчики и тестировщики, получающие чёткое ТЗ, стабильный API и быстрое исправление багов.                                                                       |
| **Регуляторы**          | Налоговая инспекция, ФСБ/ФСТЭК, Трудовая инспекция – предъявляющие требования к легитимности первичных документов, защите информации и соблюдению Трудового кодекса. |

## 2.2. Системные элементы (типовая конфигурация)

| **Компонент**                                     | **Назначение**                                                                                                                                                                                                                                                                                       |
|---------------------------------------------------|------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| **Маршрутизатор**                                 | Ядро Dockies – управляет движением задач по ролям и контролирует сроки согласования. Определяет последовательность/параллельность шагов маршрута на основе типа документа и шаблона, настроенного Администратором.                                                                                   |
| **Репозиторий документов**                        | Хранилище файлов с поддержкой версионности. Хранит историю изменений, контрольные суммы каждой версии (хеш SHA‑256) и связь версий с карточкой документа.                                                                                                                                            |
| **Модуль электронной подписи**                    | Интеграция с внешним криптопровайдером / УЦ. Поддерживает КЭП (для договоров с контрагентами и юридически значимых действий) и УНЭП (для внутренних согласований). Перед подписанием проверяет статус сертификата по протоколу OCSP.                                                                 |
| **Журнал аудита**                                 | Неизменяемая лента событий – каждая запись содержит ID события, ID пользователя, временную метку, IP‑адрес устройства, тип действия, контрольную сумму файла (если применимо). Записи связаны криптографической цепочкой. Доступен для Службы безопасности, Администратора и Юриста (при проверках). |
| **Модуль единого входа (SSO)**                    | Подсистема аутентификации, интегрированная с корпоративным AD. При входе пользователя запрашивает проверку учётных данных и получает токен с идентификатором пользователя, списком ролей, email и атрибутом manager.                                                                                 |
| **Сервис уведомлений**                            | Отправка уведомлений по email, push‑уведомлений и SMS через внешний шлюз. При недоступности основного канала переключается на альтернативный.                                                                                                                                                        |
| **Интеграция с HRIS / справочником оргструктуры** | Получает актуальную иерархию подразделений, штатное расписание, лимиты согласования по суммам и временные замещения.                                                                                                                                                                                 |
| **Интеграция с бюджетной системой / ERP**         | Получение остатков бюджетов, формирование заявок на оплату после согласования.                                                                                                                                                                                                                       |
| **Модуль выгрузки в долговременный архив**        | Передача завершённых дел во внешний архив (ECM) с сохранением всех версий файлов, подписей, метаданных карточки и ленты аудита с получением квитанции.                                                                                                                                               |
| **Интерфейс для Контрагента**                     | Личный кабинет (через оператора ЭДО) для получения подписанных сканов/электронных документов и подачи заявок.                                                                                                                                                                                        |
| **Модуль интеграции с оператором ЭДО**            | Отправка и получение юридически значимых документов через внешнего оператора электронного документооборота.                                                                                                                                                                                          |

## 2.3. Сценарий 1. Создание и согласование договора с контрагентом

### Цель
Создать проект договора с контрагентом, провести юридическую экспертизу, бизнес‑утверждение, подписать КЭП и направить контрагенту через оператора ЭДО, с контролем версий и полным аудитом.

### Условия начала
- Инициатор авторизован в Dockies через Модуль единого входа – получен токен с ролями.
- Справочник оргструктуры (HRIS) актуален – известны роли Юриста и Согласующего.
- Реквизиты контрагента проверены через систему бюджетирования / ERP – статус контрагента активен.
- Шаблон договора доступен для загрузки.

### Основной поток
1. **Инициатор** создаёт карточку договора, заполняет реквизиты (контрагент, сумма, предмет, статья расходов) и загружает файл – **Версия 1**.
2. **Маршрутизатор** обращается к Справочнику оргструктуры, определяет маршрут: первый шаг – **Юрист**.
3. **Юрист** получает уведомление через Сервис уведомлений (email и/или push-уведомление), открывает задачу, вносит правки, загружает **Версию 1.1** с пометкой *«Правки юриста»*.
    - *Репозиторий документов* сохраняет обе версии с контрольными суммами, связывая их с одной карточкой.
4. **Юрист** нажимает *«Согласовано»* – система фиксирует визу в журнале аудита и передаёт задачу **Согласующему**.
5. **Согласующий** просматривает список версий (номер версии, автор, дата, комментарий, контрольная сумма), утверждает файл и накладывает **КЭП** через Модуль электронной подписи:
    - Модуль вычисляет хеш файла, отправляет в криптопровайдер, получает подписанный хеш и штамп времени, сохраняет подпись в карточке документа.
    - Журнал аудита фиксирует: ID подписанта, контрольную сумму, серийный номер сертификата, временную метку.
6. **Модуль интеграции с оператором ЭДО** направляет подписанный документ **Контрагенту** и фиксирует в журнале аудита факт отправки и квитанцию оператора.
7. **Контрагент** подписывает встречной КЭП. Оператор ЭДО возвращает квитанцию и подписанный экземпляр. Система проверяет подпись и обновляет статус документа на **«Договор действует»**.

### Результат
Договор создан, проверен, утверждён и подписан обеими сторонами. Все версии, визы, подписи и факты отправки зафиксированы в журнале аудита с криптографической защитой.

### Исключения (нештатные ситуации)

| **№** | **Сценарий**                                                                          | **Реакция системы**                                                                                                                                 |
|-------|---------------------------------------------------------------------------------------|-----------------------------------------------------------------------------------------------------------------------------------------------------|
| 1.1   | Позиция в стоп‑листе (запрещённый тип документа / контрагент)                         | Маршрутизатор блокирует создание задачи, уведомляет Инициатора с предложением обратиться к Администратору.                                          |
| 1.2   | Конфликт версий при параллельном редактировании (два сотрудника правят один документ) | Блокировка операции вторым редактором, предложение скачать актуальную версию, объединить и загрузить новую.                                         |
| 1.3   | Потеря сессии Инициатора (разлогинило)                                                | Dockies сохраняет черновик карточки, при повторном входе предлагает продолжить.                                                                     |
| 1.4   | Оператор ЭДО недоступен                                                               | Документ переводится в статус *«Ожидает отправки»*, при восстановлении связи – автоматическая отправка. Факт перехода фиксируется в журнале аудита. |

### Режимы деградации

- **Криптопровайдер / УЦ недоступен**  
  → Документ переводится в статус *«Ожидает подписания»*, Сервис уведомлений отправляет напоминание Администратору. При восстановлении – автоматическое подписание и отправка.

- **Справочник оргструктуры / HRIS не отвечает**  
  → Маршрутизатор использует кэшированную структуру (последний успешный запрос). В журнале аудита фиксируется пометка *«Использован кэш»*. При длительной недоступности задача направляется Администратору для ручного назначения согласующего.

- **Модуль единого входа недоступен**  
  → Мобильное приложение / веб‑версия использует кэшированный токен, если сессия активна. При истечении токена – вход невозможен, отображается сообщение «Сервис аутентификации временно недоступен».

## 2.4. Сценарий 2. Служебная записка на оплату (Бухгалтер, бюджетный контроль)

### Цель
Провести служебную записку / заявку на оплату счёта через Бухгалтера, с автоматической проверкой бюджетных лимитов (интеграция с ERP) и корректности реквизитов, с контролем версий и согласованием у Согласующего.

### Условия начала
- Инициатор авторизован в Dockies.
- Справочник оргструктуры актуален – известен Согласующий (руководитель по иерархии) и Бухгалтер, ответственный за статью бюджета.
- Интеграция с ERP / бюджетной системой активна (для проверки остатков).
- Карточка документа содержит: тип *«Служебная записка на оплату»*, сумму, код ЦФО, счёт на оплату, приложенный файл счёта.

### Основной поток
1. **Инициатор** создаёт служебную записку, заполняет карточку: контрагент, сумма, код ЦФО, прикрепляет файл счёта. Маршрутизатор определяет маршрут: первый шаг – **Бухгалтер** (проверка реквизитов и бюджетного лимита).
2. **Бухгалтер** получает уведомление, открывает задачу:
    - Модуль интеграции с ERP автоматически проверяет реквизиты контрагента и остаток бюджета по коду ЦФО.
    - Если всё корректно – Бухгалтер нажимает *«Согласовано по бюджету»*, система фиксирует визу.
3. **Маршрутизатор** передаёт задачу **Согласующему** (руководителю Инициатора по иерархии HRIS).
4. **Согласующий** просматривает контекст (реквизиты, сумму, остаток бюджета, визу Бухгалтера), накладывает **УНЭП** или **КЭП** (в зависимости от суммы/политики безопасности), система фиксирует подпись в журнале аудита.
5. После подписи Маршрутизатор проверяет:
    - Если требуется дополнительное согласование (например, при сумме выше лимита руководителя) – передаёт вышестоящему Согласующему.
    - Если все согласования пройдены – обновляет статус служебной записки на **«Оплатить»** и формирует заявку на оплату в ERP (через интеграцию) или выгружает задачу Бухгалтеру для ручного запуска платежа.
6. **Инициатор** и **Бухгалтер** получают уведомление о готовности документа. **Спонсор проекта** (при необходимости) – сводку по суммам заявок.

### Результат
Служебная записка проверена по бюджетным лимитам и реквизитам, утверждена руководителем, зафиксирована в журнале аудита, передана в ERP для исполнения платежа.

### Исключения

| **№** | **Сценарий**                                  | **Реакция системы**                                                                                                                                                                                                                          |
|-------|-----------------------------------------------|----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| 2.1   | Ошибка в реквизитах контрагента (ИНН/КПП/р/с) | Бухгалтер получает задачу с пометкой «Реквизиты некорректны», вносит правки или отправляет на доработку Инициатору.                                                                                                                          |
| 2.2   | Превышение бюджетного лимита по коду ЦФО      | Задача блокируется, Бухгалтер не может согласовать, система предлагает выбрать другой ЦФО или запросить дополнительный бюджет.                                                                                                               |
| 2.3   | Смена цен/лимитов в процессе (обновилась ERP) | Dockies фиксирует цены на момент отправки служебной записки и не пересчитывает задним числом. В случае расхождения – задача направляется Администратору.                                                                                     |
| 2.4   | Отсутствие интеграции с ERP (сеть недоступна) | Модуль проверки лимитов переходит в ручной режим – Бухгалтер получает задачу с пометкой «Проверить лимиты вручную», активирует элемент «Лимит подтверждён вручную» (берёт ответственность на себя), система фиксирует факт в журнале аудита. |

### Режимы деградации

- **ERP / бюджетная система недоступна**  
  → Автоматический переход в ручной режим (п. 2.4 выше). При восстановлении связи – синхронизация и сверка ручных операций.

- **Сервис уведомлений недоступен**  
  → Push-уведомления и email не отправлены. Dockies повторяет отправку через альтернативный канал (SMS → email), но не блокирует движение задачи (пользователь может увидеть задачу, зайдя в систему).

- **Модуль единого входа недоступен**  
  → Веб‑версия использует кэшированный токен активной сессии. При истечении токена – вход невозможен, на мобильных устройствах – аналогично.

## 2.5. Сценарий 3. Регистрация входящей корреспонденции и контроль поручений (Секретариат)

### Цель
Зарегистрировать входящее письмо (бумажное или электронное), назначить ответственного (Согласующего), установить срок исполнения, контролировать выполнение поручения, зафиксировать результат в журнале аудита.

### Условия начала
- Секретарь авторизован в Dockies.
- Входящий документ поступил на бумажном носителе или по электронной почте.
- В системе настроен справочник типов входящих документов и шаблонов поручений.

### Основной поток
1. **Секретарь** создаёт карточку входящего документа, сканирует бумажный оригинал или прикрепляет электронный файл, заполняет реквизиты: отправитель, дата поступления, краткое содержание.
2. **Секретарь** назначает **Согласующего** (руководителя, ответственного за исполнение) и устанавливает срок исполнения.
3. Система генерирует **поручение** и отправляет уведомление Согласующему через Сервис уведомлений.
4. **Согласующий** получает задачу, просматривает документ, выносит резолюцию (текстовый комментарий) и/или делегирует исполнение другому сотруднику.
5. **Согласующий** отмечает поручение как исполненное (с приложением результата – файла ответа, отчёта).
6. Система фиксирует все действия в журнале аудита (регистрация, назначение, резолюция, исполнение).
7. **Секретарь** видит дашборд контроля поручений (статус, отклонение от срока). При необходимости может направить напоминание.

### Результат
Входящий документ зарегистрирован, поручение назначено и выполнено в срок, все этапы зафиксированы в аудите. Обеспечена прозрачность контроля поручений.

### Исключения

| **№** | **Сценарий**                                    | **Реакция системы**                                                                   |
|-------|-------------------------------------------------|---------------------------------------------------------------------------------------|
| 3.1   | Назначен несуществующий или уволенный сотрудник | Система блокирует назначение, предлагает актуальный список из HRIS.                   |
| 3.2   | Пропущен срок исполнения поручения              | Ежедневное автоматическое уведомление Согласующему и Секретарю до закрытия поручения. |
| 3.3   | Потеря сессии Секретаря при регистрации         | Dockies сохраняет черновик карточки, при повторном входе предлагает продолжить.       |

### Режимы деградации

- **Справочник оргструктуры / HRIS недоступен**  
  → Секретарь может вручную ввести ФИО сотрудника (но система пометит как «не верифицировано»). При восстановлении связи – проверка и автозамена на актуальный ID.

- **Сервис уведомлений недоступен**  
  → Поручение создаётся, но уведомление не отправлено. Секретарь информирует сотрудника по телефону или email вручную. Система повторяет попытку отправки каждые 15 минут.

## 2.6. Сценарий 4. Кадровая служебная записка на отпуск/премию (Отдел кадров)

### Цель
Сотрудник (Инициатор) подаёт служебную записку на отпуск или премирование. Кадровый специалист проверяет право на отпуск/премию (использованные дни, стаж, показатели), Согласующий (руководитель) утверждает, Бухгалтер (при премии) проводит начисление. Все версии и решения сохраняются в личном деле сотрудника.

### Условия начала
- Инициатор авторизован в Dockies.
- Кадровый специалист (Отдел кадров) имеет доступ к личным делам и кадровым справочникам.
- Для премии – интеграция с ERP для проверки бюджетных лимитов по фонду оплаты труда.

### Основной поток
1. **Инициатор** создаёт служебную записку типа «На отпуск» или «На премию», указывает период (для отпуска) / сумму и основание (для премии).
2. **Маршрутизатор** направляет задачу в **Отдел кадров**.
3. **Кадровый специалист** проверяет:
    - Для отпуска: остаток дней по личному делу, соблюдение графика отпусков.
    - Для премии: наличие оснований (положение о премировании, KPI).
    - При несоответствии – возвращает на доработку Инициатору с комментарием.
4. После успешной проверки кадровый специалист нажимает «Согласовано по кадрам», система передаёт задачу **Согласующему** (руководителю Инициатора).
5. **Согласующий** утверждает или отклоняет служебную записку.
6. Если документ утверждён и это премия, Маршрутизатор направляет задачу **Бухгалтеру** для расчёта и включения в зарплатную ведомость (через ERP).
7. **Бухгалтер** подтверждает начисление. Система фиксирует событие в журнале аудита и обновляет личное дело сотрудника (использованные дни отпуска, суммы премий).
8. **Инициатор** получает уведомление о результате.

### Результат
Служебная записка на отпуск или премию проверена кадрами, утверждена руководителем, при премии – передана в бухгалтерию. Изменения отражены в личном деле сотрудника.

### Исключения

| **№** | **Сценарий**                                                  | **Реакция системы**                                                                                                                           |
|-------|---------------------------------------------------------------|-----------------------------------------------------------------------------------------------------------------------------------------------|
| 4.1   | У сотрудника недостаточно дней отпуска                        | Кадровый специалист отклоняет заявку с комментарием, система уведомляет Инициатора и предлагает перенести период.                             |
| 4.2   | Для премии превышен бюджет ФОТ                                | Бухгалтер получает задачу с пометкой «Бюджет ФОТ превышен», заявка возвращается Согласующему для дополнительного согласования с финансистами. |
| 4.3   | Конфликт периодов отпуска с другим сотрудником в одном отделе | Кадровый специалист видит график отпусков, предлагает согласовать перенос.                                                                    |

### Режимы деградации

- **Интеграция с ERP для бюджета ФОТ недоступна**  
  → Бухгалтер переходит в ручной режим: проверяет лимиты в альтернативной системе (таблица Excel) и фиксирует в Dockies пометку «Лимит проверен вручную». Запись в журнале аудита.

- **Личное дело сотрудника временно недоступно**  
  → Кадровый специалист запрашивает данные из внешней кадровой системы (резервный API). При полной недоступности – заявка ставится в очередь с пометкой «Ожидание данных», Инициатор уведомляется о задержке.

## 2.7. Сценарий 5. Выгрузка данных для проверки регулятором

### Цель
Предоставить регулятору (налоговая инспекция, ФСБ/ФСТЭК, трудовая инспекция) выгрузку документов, подписей и аудиторских логов за требуемый период в соответствии с законодательством, с защитой от модификации и обеспечением конфиденциальности.

### Условия начала
- Поступил официальный запрос от регулятора (письменное требование).
- Администратор системы авторизован с правами экспорта данных.
- Включён режим аудита – все экспортные операции логируются.

### Основной поток
1. **Администратор** в специальном интерфейсе «Экспорт для регулятора» выбирает: тип документов (договоры, служебные записки, кадровые приказы и т.д.), диапазон дат, формат выгрузки (XML, JSON, структурированный PDF/A).
2. Система формирует **пакет выгрузки**, включающий:
    - Карточки документов (все версии).
    - Электронные подписи (КЭП, УНЭП) с сертификатами и штампами времени.
    - Журнал аудита по каждому документу (кто, когда, что сделал, контрольные суммы).
3. Система подписывает пакет **квалифицированной электронной подписью** Администратора (или технической учётной записи) и помещает его в защищённый архив.
4. **Администратор** передаёт архив регулятору на внешнем носителе или через защищённый канал (портал госуслуг, ВКС).
5. **Служба безопасности** получает уведомление о факте выгрузки, проверяет легитимность запроса и может наложить вето (отменить экспорт, если запрос недействителен). В случае вето – экспорт блокируется, событие фиксируется в журнале.
6. **Регулятор** принимает пакет, при необходимости проверяет подпись и целостность данных.

### Результат
Сформирована юридически значимая выгрузка, удовлетворяющая требованиям регулятора, с сохранением неотказуемости и неизменяемости. Все действия Администратора и Службы безопасности зафиксированы в аудите.

### Исключения

| **№** | **Сценарий**                                                       | **Реакция системы**                                                                                      |
|-------|--------------------------------------------------------------------|----------------------------------------------------------------------------------------------------------|
| 5.1   | Запрашиваемый период содержит документы с грифом «Особая важность» | Экспорт блокируется, требуется дополнительное разрешение от Службы безопасности.                         |
| 5.2   | Некорректный запрос (нет основания, неправильный формат)           | Администратор возвращает запрос регулятору на доработку до начала выгрузки.                              |
| 5.3   | Нет свободного места для формирования архива                       | Система уведомляет Администратора, прерывает экспорт. После освобождения места – возможность продолжить. |

### Режимы деградации

- **Журнал аудита частично недоступен (сбой БД)**  
  → Экспорт разрешён только после восстановления целостности журнала. Администратор не может выгрузить неполные данные – система блокирует до исправления.

- **Криптопровайдер для подписи пакета недоступен**  
  → Выгрузка не может быть подписана КЭП. Система ожидает восстановления, уведомляет Администратора. Возможен ручной режим: подпись на внешнем носителе (USB-токен), но это требует отдельной процедуры.
